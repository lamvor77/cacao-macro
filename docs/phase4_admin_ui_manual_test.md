# Phase 4-2 운영 관리자 UI 수동 테스트 절차

**중요**: `docs/sql/phase4_admin_rpc.sql`이 실제 Supabase 프로젝트에 적용되기 전에는
1~15번(실제 RPC 호출이 필요한 항목)을 진행할 수 없다 — 이 문서 작성 시점 기준으로는
**Fake 기반 자동화 테스트(`tests/test_operations_admin_panel.py` 등)만 가능**하며,
아래 절차는 SQL이 테스트 Supabase 프로젝트에 적용된 뒤 사용한다. 절대 운영
프로젝트에서 먼저 시도하지 않는다.

## 사전 준비

- `docs/sql/phase4_admin_rpc.sql`을 테스트 Supabase 프로젝트에 적용(`docs/sql/phase4_admin_rpc_test_plan.md` A절 계정 준비 포함).
- `.env`에 테스트 프로젝트의 `SUPABASE_URL`/`SUPABASE_ANON_KEY` 설정, `SUPABASE_ENABLED=true`.
- admin 계정 1개(approved) + editor/viewer/pending/blocked 계정 각 1개로 로그인 테스트.

## 절차

1. **admin 로그인 시 버튼 표시** — admin(approved) 계정으로 Google 로그인 → 제목표시줄에
   "운영 관리자" 버튼이 나타나는지 확인.
2. **editor/viewer 로그인 시 버튼 미표시** — editor 또는 viewer(approved) 계정으로 로그인
   → "운영 관리자" 버튼이 보이지 않는지(비활성화가 아니라 아예 없는지) 확인.
3. **pending/blocked 사용자 버튼 미표시** — pending 또는 blocked 계정으로 로그인 →
   "운영 관리자" 버튼 없음, 클라우드 상태 라벨은 기존대로 "관리자 승인 대기"/"접근 차단됨" 표시.
4. **사용자 목록 로드** — "운영 관리자" 버튼 클릭 → 새 창이 열리고 잠시 후 사용자 목록이
   자동으로 채워지는지("조회 중..." → 실제 건수 표시).
5. **검색** — 검색창에 이메일 일부 입력 후 Enter → 해당 사용자만 표시되는지. 검색어를
   지우고 다시 조회 → 전체 목록 복원되는지.
6. **role/status 필터** — 상태 필터를 "pending"으로, 역할 필터를 "editor"로 각각 바꿔보고
   콤보박스 변경 즉시 재조회되는지.
7. **pending 승인** — pending 사용자 선택 → "승인" 버튼 활성화 확인 → 클릭 → 역할 선택
   다이얼로그(viewer/editor만 표시, admin 없음)에서 editor 선택 → 승인 → 성공 메시지 →
   목록에서 해당 사용자 status가 approved로 바뀌는지.
8. **역할 변경** — approved 사용자 선택 → "역할 변경" → 다이얼로그에서 editor→admin 선택 →
   "확인" 버튼이 활성화되는지(같은 값 선택 시 비활성화되는지도 확인).
9. **admin 승격 경고** — 위 8번에서 admin을 선택하면 "이 사용자는 다른 사용자를
   승인·차단하고 역할을 변경할 수 있게 됩니다" 추가 확인 다이얼로그가 뜨는지, 취소 시
   실제로 아무 것도 바뀌지 않는지.
10. **차단** — approved 일반 사용자(자기 자신 아님) 선택 → "차단" → 확인 다이얼로그 →
    사유 입력(선택) → 차단 완료 → status가 blocked로 바뀌는지.
11. **자기 차단 방지** — 로그인한 admin 본인 계정을 목록에서 선택 → "차단" 버튼이
    비활성화되어 있는지(클릭 자체가 안 되는지).
12. **마지막 admin 보호 오류 표시** — 테스트 프로젝트에서 approved admin이 1명뿐인
    상태를 만든 뒤(다른 admin을 임시로 editor로 낮춰서) 그 마지막 admin을 다른 계정으로
    로그인해 차단/강등 시도 → "마지막 승인된 관리자는..." 한국어 오류 메시지가 표시되는지
    (SQL 원본 오류나 영어 SQLSTATE가 그대로 노출되지 않는지 확인).
13. **차단 해제 approved** — blocked 사용자 선택 → "차단 해제" → 복원 상태 "approved" →
    즉시 approved로 복원되고 기존 역할이 유지되는지.
14. **차단 해제 pending** — 다른 blocked 사용자를 "pending"으로 복원 → status가 pending으로
    바뀌는지(다시 승인이 필요한 상태가 되는지).
15. **감사로그 조회** — "감사 로그" 탭으로 이동 → 지금까지의 작업(승인/차단/역할변경/차단해제)이
    한국어 작업명("사용자 승인", "사용자 차단" 등)으로 표시되는지, 사용자 관리 탭에서
    특정 사용자를 선택 후 "이 사용자의 감사로그" 버튼으로 필터가 적용되는지, "필터 해제"로
    다시 전체가 보이는지.
16. **네트워크 오류** — 테스트 중 일시적으로 네트워크를 끊거나 `.env`의 `SUPABASE_URL`을
    잘못된 값으로 바꾼 뒤 조회/변경 시도 → 팝업에 원본 오류(SQL/토큰 등)가 아니라
    "관리자 작업을 처리하지 못했습니다..." 같은 안전한 문구만 뜨는지.
17. **권한 변경 중 관리자 창 종료** — 운영 관리자 창을 연 상태에서, 다른 세션(SQL Editor)에서
    현재 로그인 계정의 role을 admin이 아닌 값으로 바꾼 뒤, 창에서 아무 작업이나 시도 →
    권한 오류 메시지 후 창이 자동으로 닫히는지, MainWindow의 "운영 관리자" 버튼도 사라지는지.
18. **로그아웃 시 창 종료** — 운영 관리자 창을 연 상태에서 MainWindow의 "로그아웃" 클릭 →
    운영 관리자 창이 자동으로 닫히는지.
19. **창 중복 실행 방지** — "운영 관리자" 버튼을 연속으로 여러 번 클릭 → 새 창이 여러 개
    뜨지 않고 기존 창이 앞으로 나오는지(포커스 이동).
20. **메인 자동화 기능 회귀 확인** — 운영 관리자 기능과 무관하게, 기존 메시지 자동 저장/
    발송/저장된 카톡방 켜기 등이 평소대로 동작하는지(이번 변경이 기존 기능에 전혀
    영향을 주지 않았는지 최종 확인).

## 참고

- 위 1~3, 17~19번은 실제 Supabase 없이도(현재 상태로) 확인 가능한 항목이며,
  `tests/test_admin_ui_permissions.py`/`tests/test_operations_admin_panel.py`가
  이미 로직 수준에서 자동 검증하고 있다.
- 4~16번은 `docs/sql/phase4_admin_rpc.sql` 적용이 선행되어야 하는 실제 RPC 연동
  테스트다.
