# 문제 해결 (Troubleshooting)

자주 발생할 수 있는 문제와 해결 방법을 모아둔 문서. 라이선스 관련 오류는
[docs/license_deployment_guide.md](license_deployment_guide.md)의 "오류별
대응" 표를 먼저 확인한다.

## PC 프로그램

### "Realtime 연결 실패" 로그가 반복된다

- `.env`의 `SUPABASE_URL`/`SUPABASE_ANON_KEY`가 올바른지 확인한다.
- 사내망/방화벽이 WebSocket(wss://) 연결을 막고 있지 않은지 확인한다 —
  Realtime은 일반 HTTPS 요청과 별개로 WebSocket 연결이 필요하다.
- 회사 프록시 환경이라면 프록시가 WebSocket을 지원하는지 확인한다.
- 연결이 끊겨도 로컬 저장/카카오톡 발송에는 영향이 없다 — Realtime은 실시간
  반영 기능일 뿐, 저장 자체는 항상 로컬이 우선이다(오프라인 우선 원칙).

### 메시지 상태 라벨이 계속 "재연결 중"이다

- 진단정보 화면에서 Supabase 네트워크 상태를 확인한다.
- 인터넷 연결을 확인한다. 연결이 복구되면 라이브러리가 자동으로 지수
  백오프 재연결을 시도하고, 재연결되면 전체 메시지를 다시 조회해 최신
  상태로 맞춘다(별도 조치 불필요).

### 메시지 상태가 "충돌"로 표시된다

- 다른 사용자(PC 또는 모바일)가 같은 메시지를 먼저 저장했다는 뜻이다.
- 해당 메시지 칸을 다시 클릭(포커스)해 최신 내용을 확인한 뒤, 필요한 수정을
  다시 반영해서 저장한다.
- 자동으로 덮어써지지 않는다 — 반드시 사용자가 확인 후 다시 저장해야 한다.

### 발송 시작 시 "서버 확인 실패" 대화상자가 뜬다

- 정상적인 동작이다(요구사항 9절) — 인터넷 연결이 없거나 Supabase에 접근할
  수 없을 때 나타난다.
- "예"를 누르면 마지막으로 동기화된(로컬 캐시) 내용으로 발송을 시작한다.
- "아니오"를 누르면 발송을 시작하지 않는다 — 네트워크를 복구한 뒤 다시
  시도한다.

### 초기 메시지 이전 대화상자가 뜨지 않는다

- 관리자(admin) 역할로 로그인되어 있는지 확인한다(일반 직원 계정에는 뜨지
  않는다).
- 서버의 1~12번 메시지가 이미 한 번이라도 실제로 수정된 적이 있으면 뜨지
  않는다(정상 — 서버 데이터가 이미 있으므로 이전 대상이 아니다).
- `storage/cloud_sync/shared_messages_migration_state.json` 파일이 이미
  있으면 이 PC에서는 다시 묻지 않는다(완료 표시) — 파일을 삭제하면 다시
  물어보게 할 수 있지만, 일반적으로 필요하지 않다.

## 모바일

### "설정 오류" 화면만 보인다

- 배포된 모바일 웹의 `.env`(빌드 시점 `VITE_SUPABASE_URL`/
  `VITE_SUPABASE_ANON_KEY`)가 비어 있다는 뜻이다 — 배포 담당자에게 문의한다.

### Google 로그인 버튼을 눌러도 반응이 없거나 오류가 난다

- Supabase 대시보드 Authentication → URL Configuration에 모바일 웹 주소가
  Redirect URL로 등록되어 있는지 확인한다.
- 브라우저의 팝업/리디렉션 차단 설정을 확인한다.

### 로그인은 되는데 "접근할 수 없습니다" 화면만 보인다

- 관리자 승인 대기(`pending`) 또는 차단(`blocked`) 상태다 —
  [docs/mobile_message_editor.md](mobile_message_editor.md)의 "직원 계정
  승인 방법"을 참고해 관리자에게 승인을 요청한다.
- 승인되었는데도 계속 이 화면이 보이면, 역할이 `viewer`로 지정되어 있을 수
  있다(이번 스프린트에서 모바일은 `editor`/`admin`만 접근 가능) — 관리자에게
  역할 변경을 요청한다.

### 저장을 눌렀는데 "다른 사용자가 먼저 저장했습니다" 오류가 뜬다

- 정상 동작이다 — 자동으로 덮어쓰지 않는다. 화면에 뜨는 안내대로 최신
  내용을 확인한 뒤 다시 수정해서 저장한다.

### 화면이 실시간으로 갱신되지 않는다

- 화면 위쪽의 연결 상태 표시를 확인한다("오프라인"/"재연결 중"이면 네트워크
  문제).
- 새로고침 버튼을 눌러 수동으로 최신 상태를 불러올 수 있다.
- 앱을 백그라운드에 오래 뒀다가 다시 열면 자동으로 최신 상태를 다시
  불러온다(별도 조치 불필요).

## 운영자 공통

### RLS 때문에 저장이 거부된다(권한 오류)

- 계정이 `approved` 상태이고 역할이 `editor` 또는 `admin`인지 확인한다
  (`viewer`나 `pending`/`blocked`는 저장 권한이 없다 —
  `docs/sql/shared_messages_realtime.sql` 상단의 "역할 매핑" 주석 참고).
- 관리자 강제 저장이 필요하면 `admin` 계정으로 다시 시도한다(일반 저장이
  실패해도 admin은 강제 저장 경로를 쓸 수 있다 — 단, 이번 스프린트에서는
  PC/모바일 화면에 "강제 저장" 버튼을 별도로 노출하지 않았으므로, 필요 시
  해당 기능을 UI에 연결하는 후속 작업이 필요하다. 자세한 내용은 이 문서의
  마지막 절 "남아 있는 위험 또는 제한"에 준하는 완료 보고서를 참고).

### 마이그레이션 SQL 적용 후 아무 것도 안 바뀐 것처럼 보인다

- `docs/sql/shared_messages_realtime.sql`은 idempotent하게 작성되어 있어
  이미 적용된 상태에서 다시 실행해도 데이터를 덮어쓰지 않는다 — 정상이다.
- `select * from shared_messages;`로 12개 행이 있는지 직접 확인한다.
