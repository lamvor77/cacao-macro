# 수동 종단간(E2E) 테스트 계획 — PC ↔ 모바일 실시간 동기화

이 문서는 실제 Supabase 테스트 프로젝트, 실제 PC 실행, 실제 모바일 브라우저를
사용하는 수동 테스트 절차다. 자동화된 단위/정적 테스트(`python -m unittest
discover -s tests`, `npm test`)로는 검증할 수 없는 "실제 네트워크 왕복"만
다룬다 — 이 문서의 시나리오는 Claude가 자동으로 실행할 수 없다(실제 Supabase
프로젝트 접근, 실제 Google 로그인, 실제 브라우저 조작이 필요).

## 환경 준비

- **Supabase 테스트 프로젝트**: 운영 프로젝트가 아닌 별도 프로젝트를 권장한다.
  `docs/sql/shared_messages_realtime.sql`을 적용한다(`docs/database_migration.md`
  절차 참고). 적용 후 `python scripts/check_shared_messages_schema.py`로
  스키마를 확인한다.
- **PC 개발 실행**: `python main.py` (실제 EXE가 아니어도 된다 — 로직은
  동일). `.env`에 테스트 프로젝트의 `SUPABASE_URL`/`SUPABASE_ANON_KEY`를 설정.
- **모바일**: `cd mobile && npm run dev`(로컬) 또는 Vercel/Netlify 등의 Preview
  배포. `.env`에 같은 테스트 프로젝트 값을 설정.
- **계정**: 일반 직원(`editor`) 계정 2개, 관리자(`admin`) 계정 1개 — 모두
  같은 Supabase 테스트 프로젝트의 `app_users`에 승인(`approved`) 상태로 등록.
- **초기 데이터**: 마이그레이션 SQL이 1~12번 시드 행을 자동 생성한다.

## 시나리오 1 — 모바일 1번 수정 → PC 반영

- **사전 조건**: PC 프로그램 실행 중(직원 A 로그인), 모바일에서 직원 A 로그인.
- **절차**: 모바일에서 메시지 1번을 열어 내용을 수정하고 저장한다.
- **기대 결과**: PC의 메시지 1번 입력창이 자동으로 갱신되고, 상태 라벨이
  "동기화됨"으로 표시된다. PC 로그에 `message_no=1` 변경 이벤트 수신 기록이
  남는다(MESSAGE_SYNC_DEBUG=true일 때).

## 시나리오 2 — PC 2번 수정 → 모바일 반영

- **절차**: PC에서 메시지 2번을 수정하고(자동저장 디바운스 2초 대기).
- **기대 결과**: 모바일 목록 화면(해당 화면이 열려 있는 상태)의 메시지 2번
  미리보기가 자동 갱신된다. 연결 배지가 "연결됨"으로 유지된다.

## 시나리오 3 — 동시에 3번 수정 → 한쪽 충돌

- **절차**: 두 클라이언트(PC, 모바일)에서 동시에 메시지 3번 편집 화면을 열고,
  각각 다른 내용으로 거의 동시에 저장을 누른다.
- **기대 결과**: 먼저 도착한 저장 요청만 성공(revision 증가). 나중 요청은
  `REVISION_CONFLICT` 오류를 받고 "다른 사용자가 먼저 저장했습니다" 메시지를
  본다. 어느 쪽도 자동으로 덮어써지지 않는다.

## 시나리오 4~7 — PC 네트워크 단절/복구 및 재연결 복구

4. PC의 네트워크를 끊는다(Wi-Fi 끄기 등). PC 상태 라벨이 "재연결 중" 또는
   "오프라인"으로 바뀌는지 확인한다.
5. 네트워크가 끊긴 동안 모바일에서 메시지 4번을 수정하고 저장한다(모바일은
   정상 네트워크).
6. PC 네트워크를 복구한다.
7. **기대 결과**: PC가 자동으로 재연결하고("실시간 연결됨"으로 전환), 전체
   재조회가 일어나 메시지 4번이 최신 내용으로 갱신된다 — 재연결 사이에 놓친
   이벤트가 revision 비교로 복구되어야 한다.

## 시나리오 8~9 — PC 재시작 시 정합성 복구

8. PC 프로그램을 완전히 종료했다가 다시 실행한다.
9. **기대 결과**: 시작 시 1~12번 전체가 서버 최신 상태로 채워진다(Realtime
   최초 연결 시 자동 전체 재조회).

## 시나리오 10 — 모바일 백그라운드 후 복귀

- **절차**: 모바일 브라우저를 백그라운드로 보냈다가(다른 앱 전환) 몇 분 뒤
  다시 연다.
- **기대 결과**: 화면이 자동으로 최신 상태를 다시 불러온다(`visibilitychange`
  이벤트 처리).

## 시나리오 11 — 관리자 강제 저장

- **사전 조건**: 관리자 계정으로 PC 또는 모바일에 로그인. 편집 중 다른
  사용자의 변경으로 "다른 직원이 수정했습니다" 배너가 뜬 상태.
- **절차**: "내 내용으로 강제 저장" 선택 → 확인 대화상자에서 승인.
- **기대 결과**: 확인 문구("서버의 최신 메시지를 덮어씁니다...")가 정확히
  표시된다. 저장 후 서버 값이 관리자가 입력한 내용으로 바뀌고, 이력
  (`shared_message_history`)에 `changed_from='admin_force'`로 기록된다.

## 시나리오 12 — 일반 직원 강제 저장 차단

- **절차**: 일반 직원(editor) 계정으로 같은 상황을 재현한다.
- **기대 결과**: "강제 저장" 버튼/선택지 자체가 화면에 보이지 않는다(PC/모바일
  모두). 만약 API를 직접 호출해 우회를 시도해도(예: 개발자 도구로 RPC 직접
  호출) `PERMISSION_DENIED` 오류로 거부되어야 한다(서버 측 최종 방어선 확인).

## 시나리오 13~14 — 발송 직전 메시지 수정 반영

13. PC에서 자동 발송이 진행되는 그룹의 메시지(예: 1번이 포함된 Group A) 발송
    시각 직전에, 모바일에서 해당 메시지를 수정한다.
14. **기대 결과**: 실제 카카오톡 발송 내용이 수정된 최신 내용으로 나가야 한다
    (발송 직전 개별 조회가 반영됨). PC 로그에 "발송 직전 최신 내용 반영" 또는
    해당 message_no의 서버 조회 성공 기록이 남는다(본문은 남지 않음).

## 시나리오 15~16 — 서버 장애 시 발송 정책

15. `.env`에서 `MESSAGE_SEND_OFFLINE_POLICY=block`으로 설정한 상태에서
    Supabase 접근을 의도적으로 막는다(예: 방화벽 차단, 잘못된 URL 임시 설정).
    자동 발송 시각이 되었을 때: 해당 메시지가 발송되지 않고 로그에 "발송 보류"
    기록이 남아야 하며, 프로그램/스케줄러는 계속 실행 상태를 유지해야 한다
    (다음 발송 시각에 다시 시도).
16. `MESSAGE_SEND_OFFLINE_POLICY=cached`로 바꾸고 동일하게 재현한다: 이번에는
    마지막으로 알려진 캐시 내용으로 발송되고, 로그에 "캐시 내용으로 발송"
    기록이 남아야 한다.

## 시나리오 17 — 장시간 연결(8시간)

- **절차**: PC 프로그램을 8시간 이상 실행 상태로 둔다(운영시간 08~19시 범위
  안에서 테스트하면 자연스럽게 확인 가능).
- **기대 결과**: 그 사이 Realtime 연결이 끊기지 않거나, 끊겨도 자동으로
  재연결된다. 메모리 사용량이 비정상적으로 증가하지 않는다(스레드/연결 누수
  없음 — 작업 관리자로 확인).

## 시나리오 18 — PC 절전 후 복귀

- **절차**: PC를 절전 모드로 전환했다가 복귀시킨다.
- **기대 결과**: Realtime 연결이 재연결되고(라이브러리의 WebSocket이 끊겼다가
  다시 붙음), 전체 재조회로 절전 중 놓친 변경사항이 복구된다.

## 시나리오 19 — 모바일 네트워크 Wi-Fi↔LTE 전환

- **절차**: 모바일 기기에서 Wi-Fi를 끄고 LTE로 전환한다(또는 반대).
- **기대 결과**: 연결 배지가 일시적으로 "다시 연결 중"이 되었다가 "연결됨"으로
  복귀한다. 전환 중 편집하던 내용이 사라지지 않는다.

## 시나리오 20 — 프로그램 종료 후 Realtime 스레드 잔존 여부

- **절차**: PC 프로그램을 정상 종료(■ 종료 버튼 또는 창 닫기)한 직후, 작업
  관리자에서 해당 프로세스가 실제로 완전히 종료되는지 확인한다. 종료 로그에
  "Realtime 구독 해제 완료"가 남는지 확인한다.
- **기대 결과**: 프로세스가 5초 이내에 완전히 종료된다(realtime_message_sync_service.py
  의 `_STOP_TIMEOUT_SECONDS`). 좀비 프로세스가 남지 않는다.

## 완료 후 처리

이 문서의 각 시나리오를 실제로 수행한 뒤에는, 표의 "실제 결과"/"통과 여부"
칸을 채워 이 파일 자체를 갱신한다(운영팀 기록용). FAIL이 나온 항목은 별도
이슈로 기록하고 원인 분석 후 재검증한다.
